# AI 协作工作流规范

## 概述

规范团队使用 AI（Trae）的协作方式。核心机制：**AI 收到模糊请求时，展示一张结构化信息清单；用户填写完毕后 AI 一次性生成。复杂任务必须拆成多个步骤执行，每个步骤最多一轮交互。**

**核心原则**：

- AI 是执行者，不是决策者——架构、接口设计、权限逻辑由人决定
- 上下文决定质量——残缺的上下文只会产出需要大改的代码
- 一次问清，一次生成——禁止在同一步骤内来回追问，禁止在信息不足时直接生成

***

## AI 行为规则（最高优先级）

> 以下规则约束 AI 自身的行为，不依赖用户自觉遵守。

### 规则 1：识别模糊请求，触发信息清单

当用户的请求属于**较大的结构性需求**且满足以下任一条件时，**禁止直接生成代码**，必须先展示对应类型的信息清单：

- 涉及新页面、新组件、新 composable、新接口逻辑的创建
- 任务范围宏大或模糊（如："帮我做完整的登录链路"、"写个后台列表页"、"加个带表单交互的复杂弹窗"）
- 没有提供必要的上下文（如接口路径、数据结构，或没有说明参考哪个现有页面/组件）

> **例外情况（直接执行，禁止触发清单）**：
> 如果是**明确的、小型的局部修改**，AI 必须**直接执行**并生成代码，禁止用清单打扰用户。包括但不限于：
> - 纯 UI / 样式微调（如："把这个按钮改成红色"、"调一下边距"）
> - 简单的文案或配置修改
> - 修复单个明确的 Bug（如："修复控制台里的空指针报错"）
> - 明确范围的单文件小重构（如："把这个函数提炼一下"）

### 规则 2：一次性收集，不来回追问

信息清单必须一次性展示所有需要的字段。**禁止在同一步骤内分多轮追问**。
用户填写完毕后，AI 根据填写内容直接生成，不再二次确认。

### 规则 3：生成前自检

生成代码前，AI 在内部默认执行以下检查，不需要用户提醒：

- 是否遵循 `vue-auto-imports`（不要手动 import 自动注入的 API/组件）
- 请求 loading / 错误处理是否遵循 `api-integration`
- 是否用了 any，能否替换为具体类型
- 内部组件是否 PascalCase；组件库与图标组件（`a-`/`van-`/`icons-`）是否按约定使用小写前缀；无子节点组件是否允许自闭合
- 是否有事件监听需要在 onUnmounted 中清理
- 是否存在过度封装（不必要的 props、emit、computed）

### 规则 4：禁止 AI 自行决策的边界

以下内容 AI **不得自行决定**，必须由用户在信息清单中明确提供：

- 接口路径和请求参数结构
- 组件拆分方案（是拆子组件还是写在一起）
- 是用全局 Pinia store 还是组件内本地状态
- 权限校验逻辑
- 路由配置

## 信息清单模板（按任务类型）

当触发规则 1 时，AI 根据请求类型展示对应清单。

### 类型 A：新页面

```
我需要了解以下信息才能开始生成，请填写后发给我：

【页面信息】
- 路由路径：/xxx
- 页面标题：
- 参考页面：@ src/views/（填写相似页面路径，没有则填"无"）

【功能清单】
- 筛选条件：（例：姓名、状态、时间范围）
- 表格列：（例：姓名 / 角色 / 状态 / 创建时间 / 操作）
- 操作按钮：（例：新增、编辑、删除、导出）
- 弹窗表单字段：（例：姓名 name:string、状态 status:0|1）

【接口信息】（没有则填"待定"，我会留占位符）
- 列表接口：GET /api/xxx，参数：{ page, size, keyword }
- 创建接口：POST /api/xxx
- 更新接口：PUT /api/xxx/:id
- 删除接口：DELETE /api/xxx/:id
- 响应中数据字段名：data.list / data.total（或填实际字段名）

【特殊要求】（没有则填"无"）
-
```

### 类型 B：新组件

```
我需要了解以下信息，请填写后发给我：

【组件信息】
- 组件名（PascalCase）：
- 放在哪个目录：src/components/ 还是 src/views/xxx/components/
- 参考组件：@ src/（有则填路径，没有则填"无"）

【Props 设计】（列出字段名、类型、是否必填、默认值）
- 例：list: UserItem[]（必填）
- 例：loading?: boolean（默认 false）

【Emits 事件】（有则列出，没有则填"无"）
- 例：select: [item: UserItem]

【组件功能描述】
-

【不需要的功能】（明确排除，防止过度生成）
-
```

### 类型 C：Bug 修复

```
请提供以下信息：

【问题描述】
- 现象：（看到了什么）
- 复现步骤：
- 期望行为：

【相关文件】
- @ 出问题的组件文件
- @ 相关 composable 或 store（如有）

【已排查的范围】（已确认没问题的部分）
-

【错误信息】（控制台报错 / 网络请求异常，粘贴完整信息）
-
```

### 类型 D：功能改造 / 重构

```
请提供以下信息：

【改造目标文件】
- @ 需要改造的文件

【当前问题】
- 现在的行为是：
- 期望改成：

【改造范围】
- 只改这一个文件，还是需要同步修改其他文件？
- 是否影响其他组件调用这个组件/函数？

【不允许改动的部分】（明确保留的逻辑）
-
```

## 任务拆解规范（人在提需求前做）

> 以下是给开发者看的规范——用 AI 前先自己拆解，再发给 AI 执行。

### 拆解顺序

```
1. TypeScript 类型定义（先定数据结构）
      ↓
2. API 常量 / 函数声明（明确接口路径）
      ↓
3. Composable（列表分页、表单逻辑等独立逻辑）
      ↓
4. 子组件（筛选面板、弹窗表单）
      ↓
5. 页面组件（组装上面的产物）
```

每一步单独发给 AI，按“清单 → 填写 → 产出”完成后再进入下一步。**禁止在信息不完整时一次性生成整页。**

### 判断用 AI 还是自己写

| 场景                    | 推荐方式         |
| --------------------- | ------------ |
| 标准列表页 / 弹窗表单          | AI 生成（结构固定）  |
| TypeScript 类型（有接口文档）  | AI 生成        |
| 已有相似组件，要生成新的          | AI 参考生成      |
| UnoCSS 类名组合 / Less 样式 | AI 生成        |
| 核心业务判断逻辑              | 自己写，AI 只补样板  |
| 组件拆分方案决策              | 自己决定，AI 执行   |
| 权限 / 安全相关逻辑           | 自己写，AI 审查    |
| Pinia store 结构设计      | 自己设计，AI 生成代码 |

## AI 生成代码 Review 清单

生成后必须过一遍，这些是 AI 最高频的失误点：

### 快速扫描（30 秒内）

- [ ] 是否遵循 `vue-auto-imports`（不要手动 import 自动注入的 API/组件）
- [ ] 请求 loading 是否遵循 `api-integration`（必须在 `finally` 中关闭）
- [ ] 有无 `any` 类型，能否换成具体类型
- [ ] 内部组件是否 PascalCase；组件库与图标组件（`a-`/`van-`/`icons-`）是否按约定使用小写前缀；无子节点组件是否允许自闭合
- [ ] `v-for` 是否有 `key`，是否与 `v-if` 在同一元素上
- [ ] 有无 `addEventListener` 没有对应 `onUnmounted` 清理

### 深度检查（功能复杂时）

- [ ] AI 是否自行定义了接口路径（应由人提供）
- [ ] 是否出现了不在项目依赖中的第三方库
- [ ] Props 是否超出当前需求（过度封装）
- [ ] 是否有超过 20 行的函数需要拆分
- [ ] script setup 代码顺序是否符合规范（类型→Props/Emits→响应式→computed→composable→方法→生命周期）

***

## 跑偏识别与止损

**以下信号说明 AI 跑偏了，停止追加需求，重新发起：**

- 生成代码量明显超出预期
- 出现项目中不使用的工具函数或库（如 axios 被换成 fetch）
- 违反了项目规范文档中的约束（参考 `skills/` 下对应规范，如 `css-style-conventions`、`vue-template-conventions` 等；例如用了 `/deep/`、`==`、魔法 z-index）
- 你需要大量修改才能让代码可用（超过 30% 需要改动）

**止损方式：**

```
不要在原有上下文继续追加修正——残缺的上下文会让 AI 越改越偏。
新开一个对话，重新填写信息清单，从头生成。
```
